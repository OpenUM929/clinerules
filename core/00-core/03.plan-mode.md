# 📌 Plan Mode 규칙

## Plan Mode이란?

- Plan mode에서는 **읽기만**
- 실행은 "수행"이라고 **명시적으로 요청**할 때까지 대기

## 계획서 작성 규칙

Plan Mode에서 사용자가 계획서(또는 설계 문서, 상세 계획) 작성을 요청한 경우:

1. **저장 위치**: `C:\Users\Administrator\.claude\plans\` (planning 폴더)
2. **파일명 형식**: `작업명_YYMMDD-HHMM-NN.md`
   - `작업명`: 하이픈(-)으로 연결된 영문 소문자 (예: `test-menu-integration`, `profanity-filter-redesign`)
   - `YYMMDD`: 작성일 (예: `260602`)
   - `HHMM`: 작성 시각 (예: `1415`)
   - `NN`: 당일 동일 작업의 순번 (01, 02, ...)
   - 예: `test-menu-integration_260602-1415-01.md`
3. **파일 내용**: 반드시 마크다운 형식. 최상단에 제목, 작성일시, 작업 유형, 상태(Pending/Approved/In Progress) 기재
4. **승인 절차**: 사용자가 "수행"으로 명시적 요청 전까지는 파일 시스템에 **저장만** 하고 실제 코드 변경은 하지 않음
5. **사용자 알림**: 계획서를 사용자에게 안내할 때, 대화창 마지막에 반드시 **파일 전체 경로**와 **파일명**을 함께 출력
   - 예: `계획서 저장 위치: C:\Users\Administrator\.claude\plans\test-menu-integration_260602-1415-01.md`
6. **계획 문서 개정 (Revision)**: 계획서 기반으로 작업 중 추가 수정/보완이 발생할 경우, 원본 계획 문서와 **동일한 이름의 폴더**를 `C:\Users\Administrator\.claude\plans\` 아래에 생성한다. 개정 문서는 해당 폴더 안에 `rev-NN` 접미사를 붙여 저장한다. (예: `fix_plan_methods_used_tracking_rev-01.md`). 최종 완료 보고서는 동일 폴더에 `완료보고_YYMMDD-HHMM.md` 형식으로 저장한다.
7. **계획 문서 기반 테스트 및 결과 관리**: 계획서 기반으로 구현 시 테스트가 필요한 경우, 해당 계획 문서 폴더 안에 `test/` 하위 폴더를 생성하여 테스트 코드 파일을 저장한다. 테스트 완료 보고서 및 리포트는 동일 폴더 안에 `result/` 하위 폴더를 생성하여 저장한다.
   - 예시 구조:
     ```
     plans/
     └── fix_plan_methods_used_tracking/
         ├── fix_plan_methods_used_tracking_rev-01.md
         ├── test/
         │   └── test_profanity_gap_fix.py
         └── result/
             └── 완료보고_260602-1430.md
     ```

## Git 커밋/푸시 시 `.clinerules` 지침 파일 취급 규칙

> ⚠️ **협업 보존 규칙**: 다른 팀원이 `.clinerules`를 동시에 작업할 수 있으므로, 반드시 아래 규칙을 준수한다.

1. **교체 금지**: `.clinerules/` 하위 파일을 커밋할 때, **파일 전체를 교체(replace/overwrite)**하지 않는다.
2. **차이 확인**: 푸시 전 `git diff HEAD~1 -- .clinerules/` 또는 `git show origin/master:.clinerules/파일명`으로 원격 내용과 비교한다.
3. **추가 방식**: 기존 내용이 유지된 상태에서, **우리가 추가한 항목만 Append** 하는 방식으로 커밋한다.
4. **삭제 방지**: 기존 지침의 섹션, 규칙, 문맥이 **삭제되거나 누락되지 않도록**, `git diff --cached -- .clinerules/`로 변경 내역을 반드시 검토한다.
5. **충돌 대응**: 원격 파일과 로컬 파일이 다를 경우, `git merge` 방식으로 양쪽 내용을 병합한 후 커밋한다. 병합 불가 시 사용자에게 보고한다.
