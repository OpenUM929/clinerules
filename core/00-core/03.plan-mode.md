# 📌 Plan Mode 규칙

## Plan Mode이란?

- Plan mode에서는 **읽기만**
- 실행은 "수행"이라고 **명시적으로 요청**할 때까지 대기

## 계획서 작성 규칙

Plan Mode에서 사용자가 계획서(또는 설계 문서, 상세 계획) 작성을 요청한 경우:

1. **저장 위치**: 프로젝트 루트 기준 `wordcloud_project/plans/YYYY/` 연도별 폴더 아래, **작업별 하위 폴더**를 생성하여 저장한다.
   - 연도별 폴더명: `YYYY/` (예: `2026/`)
   - 작업 폴더명: `MMDD_NN_작업명/` (파일명과 동일)
   - 모든 계획 문서는 해당 작업 폴더 내부에 저장
   - 예: `D:\dev\wordcloud\wordcloud_project\plans\2026\0605_03_deploy-retry-cascade\`

2. **파일명 형식**: `MMDD_NN_작업명.md`
   - `MMDD`: 작성일 월일 (예: `0605`)
   - `NN`: 당일 동일 작업의 순번 (01, 02, ...)
   - `작업명`: 하이픈(-)으로 연결된 영문 소문자, **최대 20자**
   - 예: `0605_03_deploy-retry-cascade.md`
   - 파일명 앞 날짜+순번으로 탐색기/터미널 정렬 시 최신 계획이 자동으로 하단에 위치

3. **파일 내용**: 반드시 마크다운 형식. 최상단에 제목, 작성일시, 작업 유형, 상태 기재

4. **승인 절차**: 사용자가 "수행"으로 명시적 요청 전까지는 파일 시스템에 **저장만** 하고 실제 코드 변경은 하지 않음

5. **사용자 알림**: 계획서를 사용자에게 안내할 때, 대화창 마지막에 반드시 **파일 전체 경로**와 **파일명**을 함께 출력
   - 예: `계획서 저장 위치: D:\dev\wordcloud\wordcloud_project\plans\2026\0605_03_deploy-retry-cascade\0605_03_deploy-retry-cascade.md`

6. **상태 관리**: 계획서 상태는 **단일 파일 내 인라인으로 관리**한다. 별도 복사본 파일을 생성하지 않는다.
   - 상태 약어:
     - `Todo`: 대기/예정 → Kanban **Todo**
     - `Doing`: 분석/검토 진행 중 → Kanban **Doing**
     - `Pre-Done`: 구현·단위검증 완료, 최종 점검 대기 → Kanban **Pre-Done**
     - `Done`: 작업 완료 — 실제 구현/적용 완료 → Kanban **Done**
     - `Hold`: 의도적 미적용·결정 대기로 **종결**한 항목 — 완료가 아닌 보류 종결. `Done`과 명확히 구분
     - `Drop`: 잘못 작성되었거나 전제가 틀려 **폐기**한 계획서 — 추진하지 않음. `Hold`(나중에 재개 가능)와 달리 되살리지 않음
   - 상태 흐름: `Todo` → `Doing` → `Pre-Done` → `Done`
   - 상태는 문서 최상단 메타 블록에 한 줄로 표기:
     - 예: `> 상태: Todo | 작성일: 2026-06-05`
     - 예: `> 상태: Done | 완료일: 2026-06-05`
     - 예: `> 상태: Hold — 사유 | 작성일: 2026-06-05`
     - 예: `> 상태: Drop — 사유 | 작성일: 2026-06-05`

7. **수정 이력 관리 (인라인 히스토리)**: 계획서에 수정/재검토 사항이 발생한 경우 별도 파일 생성 없이 단일 파일 내에서 관리한다.
   - 문서 최상단(메타 블록 바로 아래)에 `## 수정 이력` 섹션을 두고 **요약 테이블**로만 기록한다.
   - 이전 수정 요청의 상세 내용은 요약 테이블 한 줄로 압축하고, 문서 내 해당 섹션을 최신 내용으로 직접 교체한다 (이전 전체 내용 복사 금지).
   - 현재 수정 요청 상세 내용은 문서 내 별도 섹션으로 전체 기록한다.
    - 완료 시 수정 이력 섹션은 그대로 보존하고 상태만 `Done`으로 변경한다.

   ```markdown
   ## 수정 이력

   | 날짜 | 변경 섹션 | 변경 요약 |
   |------|-----------|-----------|
   | 2026-06-05 | §3 구현 방식 | retry API 엔드포인트 경로 변경 |
   | 2026-06-05 | §5 결함 목록 | 결함 2 (sessionId 누락) 추가 |
   ```

8. **연도별 인덱스 (`_index.md`) 관리**:
   - 연도별 폴더(`YYYY/`)에는 반드시 `_index.md` 파일을 유지한다.
   - `_index.md`는 탐색기 정렬 상단 고정을 위해 언더스코어 prefix를 사용한다.
   - **계획서 작성, 상태 변경(Todo→Pre-Done→Done), 수정 이력 추가 시 반드시 `_index.md`를 동시에 업데이트한다.**
   - 월별 폴더 최초 생성 시 `_index.md`도 함께 생성한다.

     ```markdown
    # 2026 계획서 인덱스

    | 계획서 | 작업 요약 | 상태 | 작성일 |
    |--------|-----------|------|--------|
     | 0605_03_deploy-retry-cascade | 실패 태스크 cascade retry | Todo | 2026-06-05 |
     | 0605_02_deploy-ux-enhance | 저장 완료 후 UX 개선 | Done | 2026-06-05 |
     | 0602_01_profanity-eng-fix | 불용어 영문 처리 개선 | Done | 2026-06-02 |
    ```

    **상태 컬럼 규칙 (Kanban 파싱 일관성)**:
    - 상태값은 **약어만** 사용: `Todo`(대기/예정) / `Doing`(분석중) / `Pre-Done`(최종 점검 대기) / `Done`(완료) / `Hold`(보류) / `Drop`(폐기)
    - `Hold`는 "구현하지 않기로 결정했거나 외부 결정·승인 대기"인 항목에 사용하며, **실제 구현 완료(`Done`)와 반드시 구분**한다. Kanban 보드의 `📌 Hold` 컬럼으로 분류된다.
    - `Drop`은 "잘못 작성되었거나 전제가 틀려 추진하지 않는" 계획서에 사용하며, **다시 되살리지 않는다는 점에서 `Hold`와 구분**한다. Kanban 보드의 `🗑️ Drop` 컬럼으로 분류된다.
    - Emoji(`✅`, `🔄`, `🔶`, `📋`, `📌`, `🗑️`)를 상태 컬럼에 **절대 포함 금지** — Kanban 보드의 UI 레이어가 자동 표시
    - 상태 컬럼 이외의 컬럼(계획서, 작업 요약, 작성일)도 emoji 사용 금지
    - 위 6개(`Todo`, `Doing`, `Pre-Done`, `Done`, `Hold`, `Drop`) 외 미등록 토큰을 쓰면 해당 행이 Kanban 파서(`src/routes/plans_routes.py` `TABLE_RE`)에 매칭되지 않아 **보드에서 누락**된다 → 반드시 등록된 약어만 사용

9. **연도 폴더 기준**: 계획서가 속하는 연도는 **작성일** 기준으로 고정한다. 2026년에 작성해 2027년에 완료되어도 `2026/`에 유지한다.

10. **계획 문서 기반 테스트 및 결과 관리**: 계획서 기반으로 구현 시 테스트가 필요한 경우, 해당 작업 폴더 안에 `test/` 하위 폴더를 생성하여 테스트 코드 파일을 저장한다. 테스트 완료 보고서 및 리포트는 동일 폴더 안에 `result/` 하위 폴더를 생성하여 저장한다.

11. **함수/클래스 정보 확인 규칙 (예측 금지)**:
    - 계획서에 언급하는 모든 함수, 클래스, 메서드, 변수명은 **반드시 실제 코드베이스에서 검색/확인**한 후에만 작성한다.
    - 파일명을 보고 "있을 것 같다" 또는 "다른 프로젝트에서 이렇게 했으니 여기도 있을 것" 같은 **추측(predict) 금지**.
    - 확인 방법: `grep` 도구로 함수명 검색, `read` 도구로 파일 직접 열어 확인.
    - 존재 여부만 확인하지 말고, **실제 함수 시그니처, 반환값, 동작 방식**까지 읽고 기술한다.
    - 함수가 존재하지 않을 경우: "해당 함수는 현재 코드베이스에 존재하지 않음 — 신규 생성 필요"라고 명시한다.

12. **DB/경로/설정 정보 확인 규칙**:
    - DB 경로, 초기화 위치, 설정 파일 위치 등 인프라 정보는 **실제 코드에서 경로 문자열을 검색**하여 확인한다.
    - "아마도 ~에 있을 것" 같은 표현 금지. 정확한 파일 경로와 코드 인용을 함께 제시한다.
    - 예시 구조:
       ```
        plans/
        └── 2026/
            ├── _index.md
            └── 0605_03_deploy-retry-cascade/
                ├── 0605_03_deploy-retry-cascade.md
                ├── test/
                │   └── test_retry_cascade.py
                └── result/
                    └── 완료보고_260605-1500.md
       ```

## 계획서 표준 템플릿 참조 규칙

1. **템플릿 필수 참조**: 계획서 작성 시 반드시 아래 작업 유형별 표준 템플릿을 참조하여 섹션을 구성한다.
2. **템플릿 위치**: 본 파일(`03.plan-mode.md`)과 동일 폴더(`00-core/`)의 `03-plan-mode/` 하위에 있다.
3. **참조 순서**: `README.md`(공통 양식 + 유형 선택 기준)를 먼저 읽고, 해당 유형 템플릿 파일을 추가로 참조한다.
4. **필수 섹션 누락 금지**: 템플릿에 명시된 필수 섹션이 계획서에 없으면 지침 위반이다.

| 작업 유형 | 템플릿 파일 | 설명 |
|-----------|-------------|------|
| 버그 수정/핫픽스 | `type-a-bugfix.md` | 문제→원인→수정→롤백 |
| 기능 개선/신규 기능 | `type-b-feature.md` | 요구사항→분석→구현→테스트 |
| 설계/아키텍처/데이터셋 | `type-c-design.md` | 목표→원칙→스키마→로드맵 |
| 리팩토링/성능 개선 | `type-d-refactor.md` | 배경→코드 분석→변경→효과 |
| DB 마이그레이션 | `type-e-migration.md` | 현황→스키마→전략→검증 |
| 복구/롤백 | `type-f-recovery.md` | 사고→범위→복구→승인 |

## Git 커밋/푸시 시 `.clinerules` 지침 파일 취급 규칙

> ⚠️ **협업 보존 규칙**: 다른 팀원이 `.clinerules`를 동시에 작업할 수 있으므로, 반드시 아래 규칙을 준수한다.

1. **교체 금지**: `.clinerules/` 하위 파일을 커밋할 때, **파일 전체를 교체(replace/overwrite)**하지 않는다.
2. **차이 확인**: 푸시 전 `git diff HEAD~1 -- .clinerules/` 또는 `git show origin/master:.clinerules/파일명`으로 원격 내용과 비교한다.
3. **추가 방식**: 기존 내용이 유지된 상태에서, **우리가 추가한 항목만 Append** 하는 방식으로 커밋한다.
4. **삭제 방지**: 기존 지침의 섹션, 규칙, 문맥이 **삭제되거나 누락되지 않도록**, `git diff --cached -- .clinerules/`로 변경 내역을 반드시 검토한다.
5. **충돌 대응**: 원격 파일과 로컬 파일이 다를 경우, `git merge` 방식으로 양쪽 내용을 병합한 후 커밋한다. 병합 불가 시 사용자에게 보고한다.
